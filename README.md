# **[프로젝트 1] 자동화 도구 비교 구현**

## 1. 과제 개요

### **1.1 목적**

- 데이터 입력 시 발생하는 단순 반복적인 확인 및 통지 업무를 효율화하기 위해 노코드 자동화 툴을 활용한 데이터 연동 시스템을 구축함

### **1.2 시나리오**

- 구글 스프레드시트에 학생의 이름과 성적이 입력되면, 자동화 봇이 이를 실시간으로 감시한다. 성적 기준에 따라 80점 이상(합격)과 80점 미만(미달)으로 데이터를 조건 분기하여, 메신저(Discord)의 독립된 채널별로 자동 알림을 전송하는 시스템을 구현함

### 1.3 노코드 자동화의 탄생 배경

노코드 자동화 툴은 과거 개발자들이 직접 코드로 구축하던 API 연동 프로그램(백엔드 개발)과 24시간 서버 인프라를 통째로 대체하기 위해 탄생함

1. 과거의 수동 API 연동 방식 (백엔드 개발):
    - API 연구: 구글과 디스코드 등 서로 다른 서비스의 API 공식 문서를 분석해야 함
    - 백엔드 코딩: Python, Node.js 등 프로그래밍 언어로 두 앱을 잇는 연결 프로그램을 직접 코딩함
    - 서버 인프라 구축: 작성한 코드가 24시간 끊김 없이 작동할 수 있도록 별도의 물리 서버나 클라우드 환경을 대여하고 상시 모니터링해야 함
2. 노코드 자동화의 혁신:
    - Make와 Zapier 같은 노코드 툴은 복잡한 API 코딩 + 백엔드 개발 + 서버 관리 과정을 시각적인 UI 맵으로 대체
    - 과거 개발자가 일주일 이상 매달려야 했던 시스템 인프라 연동 작업을 마우스 드래그와 몇 번의 클릭만으로 10분 만에 안정적으로 구현할 수 있게 됨

## 2. 사용한 기술 및 툴 아키텍처

글로벌 대표 노코드 RPA 플랫폼인 Make와 Zapier 두 가지 활용하여 시스템 비교

- **Trigger (인입):** Google Sheets
- **Filter / Router (분기):** Make의 Router / Zapier의 Paths
- **Action (발송):** Discord

#### 개념 정리

- Trigger: 방아쇠라는 뜻으로, 자동화를 시작하게 만드는 특정 조건, 이벤트 또는 신호
- 조건 분기(Filter/Router): 유입된 데이터의 특정 값을 실시간으로 판별하여 설정된 기준에 부합하는 경로로만 데이터가 흐르도록 제어하고 필터링하는 것
- Action: 트리거가 감지되어 방아쇠가 당겨진 후 실제로 수행되는 작업이나 결과

## 5. Make vs Zapier 차이점

두 가지 도구의 기능, 인터페이스, 라이선스 정책 등을 다각도로 비교 분석함

### 5.1 5대 핵심 지표 비교 분석 표

| 비교 항목 | **Make**  | **Zapier**  |
| --- | --- | --- |
| **UI/UX 및 직관성** | 원형 모듈과 시각적 노드 연결 방식을 채택하여, Router 구조를 2차원 맵 형태로 한눈에 보기 편함 | 세로형 스텝 바이 스텝 방식을 채택하여 직관적이고 직해성이 높지만 Paths가 많아지면 화면 스크롤이 길어짐 |
| **설정 난이도** | 데이터 맵핑 및 함수 활용 시 문법적 이해가 다소 필요하여 초반 진입 장벽이 있으나, 익숙해지면 자유도가 높음 | 자연어 기반의 가이드와 친절한 UI 툴팁을 제공하여 노코드 입문자가 가장 빠르게 워크플로우를 설계할 수 있을 만큼 난이도가 낮음 |
| **연동 서비스 범위** | 글로벌 주요 SaaS 서비스들과 대부분 연동되나, 간혹 최신 API 반영이나 마이너한 앱 연동 시 데이터 구조를 직접 파악해야 할 때가 있음 | 현존하는 노코드 RPA 툴 중 가장 방대한 서드파티 앱 연동 생태계를 보유하고 있어, 대부분의 마이너한 서비스도 검색만으로 쉽게 연동됨 |
| **무료 플랜 범위** | 무료 플랜에서도 무제한 시나리오 생성 및 다중 분기(Router) 기능을 제한 없이 사용할 수 있어 실습 및 소규모 테스트에 매우 유리함 | 무료 플랜의 제약이 큰 편이며, 본 과제의 핵심 기능인 조건 분기(Paths) 기능은 유료 플랜(또는 Trial 기간)에서만 제한적으로 제공됨 |
| **실행 로그 확인 방식** | 시나리오 실행 시 각 모듈 우측 상단에 숫자가 적힌 '버블(Bubble)'이 생성되어, 어떤 데이터가 오갔는지 디버깅하고 팩트를 추적하기 매우 직관적임 | Zap History 메뉴를 통해 실행 시간별 성공/실패 여부와 인/아웃풋 데이터를 텍스트 로그 형태로 깔끔하게 정돈하여 제공 |

**▲ 표 2: make와 zapier의 비교 분석**

### 5.2 도구별 장단점 및 구현 과정 요약

#### Make

- **구현 과정 요약:** 구글 시트 트리거 모듈 설정 ➔ Router 모듈을 통한 물리적 선 연결 분기 ➔ 각 연결선에 Filter 설정(성적 조건 입력) ➔ 디스코드 액션 모듈 연동 및 구글 시트 데이터 변수 매핑.
- **장점:** 강력한 무료 플랜 혜택, 복잡한 다중 분기 및 대량 데이터 처리에 최적화된 비주얼 디버깅 환경.
- **단점:** 데이터 맵핑 창 및 타입 변환 로직이 초보자에게는 다소 직관적이지 못함.

#### Zapier

- **구현 과정 요약:** 구글 시트 트리거 설정 ➔ 유료 액션인 Paths 모듈 추가 ➔ Path A와 Path B 내부에 조건 수식(Greater than, Less than 등) 지정 ➔ 디스코드 발송 모듈 연동 및 알약 모양의 데이터 칩 매핑.
- **장점:** 극도의 단순함과 사용자 친화적 UI, 이상/이하 연산자가 없을 때 텍스트 가이드를 통해 직관적인 우회 경로를 찾기 쉬움.
- **단점:** Paths(분기) 기능이 유료 등급에 묶여 있어 유지 비용이 높고, 화면이 세로로만 전개되어 대규모 아키텍처 설계 시 복잡도가 올라감.

### 5.3 어떤 상황에서 어떤 도구가 적합한지에 대한 분석 의견

- **Make가 적합한 상황 (비용 효율성 & 복잡성 중심):**
    - 초기 구축 비용을 최소화하면서 복잡한 다단계 조건 분기를 구현해야 할 때 적합함
    - 시각적인 맵 구조를 보며 직관적인 디버깅이 필요한 개발자 성향의 기획자, 스타트업 연구 환경에 적합함
- **Zapier가 적합한 상황 (속도 & 편의성 중심):**
    - RPA를 처음 접하는 비개발자 조직에서 학습 비용을 최소화하고  몇 분 만에 안정적인 자동화 연동 프로토타입을 뽑아내야 할 때 적합함
    - 연동하고자 하는 사내 툴의 종류가 매우 다양하고 마이너하여 시스템 간 인프라 호환성이 최우선 과제인 비즈니스 실무 운영 환경에 적합함

## 6. 결론 및 느낀 점

- RPA 작동 메커니즘의 이해: Make와 Zapier 모두 무료 플랜 기준 약 15분의 주기를 두고 구글 시트를 감시한다는 작동 스케줄링의 원리를 파악할 수 있었음
- 과거에는 앱과 앱을 연결하려면 반드시 프로그래밍 언어로 코딩을 해야 했는데, 노코드 툴이 나오면서 이 과정을 마우스 클릭 몇 번으로 끝낼 수 있게 바꾸어 개발자가 직접 코드로 짜던 API 연동 프로그램(백엔드 코드)을 대체하기 위해 노코드
- 제약 조건 해결 능력 배양: 툴 내에 원하는 연산자(이상/이하)가 제공되지 않더라도, or (조건 결합) 연산이나 초과/미만 등의 논리 구조를 활용해 우회 구현하는 노코드 아키텍처 설계 능력을 기를 수 있었던 유의미한 실습이었음

# [프로젝트 2] 자유 주제 자동화 설계 및 구현

## 1. 기획 배경 및 문제 정의

- 기획 배경 및 반복 업무 정의: 일상 속에서 자기관리를 위해 매일 반복되는 운동 루틴과 코디세이(공부) 일정을 소화하고 있으나, 이를 플래너나 노션에 매번 수동으로 기록하는 과정에서 심각한 번거로움과 인지적 피로감을 느낌
- 문제 정의: 이러한 수동 기록 방식은 피로 누적으로 인해 기록 누락이 잦아지고, 학습 및 운동의 연속성을 시각적으로 즉각 확인하기 어려워 장기적인 자기관리 효율성이 저하됨
- 구축 목적 및 요구사항 충족:
    - 구글 캘린더 일정에 [완료] 머리말을 붙이는 단 한 번의 조작만으로 노션 데이터베이스에 학습/운동 이력이 실시간 자동 축적되는 파이프라인을 구축하고자 함
    - Make의 Router 기능을 활용하여 완료된 루틴의 성격(운동/공부)에 따라 디스코드의 전용 채널(#오운완, #열공인증)로 맞춤형 동적 응원 메시지를 발송함으로써 성취감을 고취하고 생산성을 극대화하는 것을 목적

## 2. 사용 기술 및 도구 선정 이유

본 프로젝트에서는 프로젝트 1을 바탕으로 **Make**를 최종 자동화 툴로 선정

- 조건 분기(Router)의 비용 효율성: 본 시나리오의 핵심은 하나의 캘린더 공급원에서 운동과 학습 경로를 정교하게 찢어주는 다중 분기 로직인데 경쟁 도구인 Zapier는 다중 분기(Paths) 기능이 유료 플랜에 묶여 있어 과금 리스크가 있는 반면, Make는 핵심 기능인 Router를 무료 플랜에서도 개수 제한 없이 완전히 무료로 제공하므로 비용 효율성이 좋음
- 시각적 직관성 및 디버깅 유리: Google Calendar ➔ Notion ➔ Router ➔ Discord (2개 채널)로 이어지는 다중 연동 구조를 2차원 캔버스 형태의 노드로 한눈에 조망할 수 있습니다. 각 모듈의 데이터 흐름을 시각적으로 추적하기 쉬워 초기 빌드 및 디버깅 난이도가 낮음
- 풍부한 텍스트 연산자 제공: 캘린더 제목에서 특정 키워드를 필터링할 때, 대소문자나 띄어쓰기 문제를 우회할 수 있는 Contains (case-insensitive) 같은 강력하고 직관적인 텍스트 연산자를 기본 제공하여 조건부 필터링 레이어를 견고하게 구축할 수 있음

# 보너스 과제 1 – AI 연동 Action 추가

- 워크플로우에 생성형 AI(예: ChatGPT, Claude 등)를 Action으로 추가해 텍스트(요약/분류/템플릿 메시지)를 자동 생성

‘[프로젝트 2]자유 주제 자동화 설계 및 구현’에서 성공적으로 구현한 후, 시스템의 피드백 품질을 한 단계 높이기 위해 에서 보너스 과제 1(생성형 AI 연동 Action 추가)을 수행하여 파이프라인을 최종 고도화하였음

## 1. 고도화 배경 및 목적

- 기존 시스템의 한계: 기존 구조에서는 디스코드로 발송되는 알림이 고정된 템플릿 문구(오늘도 오운완 완료!, 오늘도 열공 완료!)로만 출력되었는데, 이러한 정형화된 메시지는 시스템을 장기적으로 사용할 때 성취감 자극 및 자기관리 지속을 유도하는 피드백 효과가 점차 반감된다는 한계가 있었음
- 고도화 목적: 노션 DB 적재와 라우터 분기 사이에 OpenAI(ChatGPT) 모듈을 에이전트로 배치 후 당일 수행한 일정 제목을 AI가 실시간으로 분석하여, 맥락에 맞는 '동적 맞춤형 응원 메시지'를 생성한 뒤 디스코드로 발송하도록 시스템 업그레이드하기

## 2. 확장된 시스템 아키텍처 및 설정 과정

AI 기능을 추가하기 위해 Make 워크플로우를 다음과 같이 확장 설계함

```
[확장 아키텍처]
Google Calendar (Trigger) ➔ Notion DB (Action 1) ➔ OpenAI ChatGPT (Action 2: 맞춤형 메시지 동적 생성) ➔ Make Router (Filter) ➔ Discord #오운완 / #열공인증 (Action 3 & 4)
```

1. OpenAI 모듈 삽입: Notion 모듈 바로 다음에 OpenAI: Create a Chat Completion 액션을 징검다리처럼 배치함 (모델: gpt-4o-mini, api 비용: 세금 포함 5.5달러)
2. 프롬프트 엔지니어링 (System Prompt 역할 부여):
    - System 역할 정의: "너는 자기관리 루틴을 응원해 주는 유능하고 친절한 AI 비서야. 사용자가 완료한 일정 제목을 주면, 그 일정에 딱 맞는 50자 내외의 짧고 유쾌한 디스코드 맞춤형 응원 메시지를 작성해 줘. 이모티콘을 적절히 섞어 쓰고, 존댓말로 작성해 줘. 다른 설명은 생략하고 오직 생성된 응원 메시지 본문만 출력해."
    - User 데이터 매핑: 구글 캘린더의 2. Summary (일정 제목 데이터 칩)를 주입하여 AI가 매번 다른 일정을 분석하도록 설계했습니다.
3. 디스코드 매핑 수정: 라우터 분기 이후, 디스코드 모듈의 Message 필드에 수동으로 입력했던 고정 텍스트를 지우고, OpenAI 모듈이 뱉어낸 동적 결과값 칩 ‘Output[]: Content[]: Text’를 매핑

## **5. OpenAI API 사용에 따른 유료 크레딧 전환 소명 및 무료 대안 검토**

### **5.1 유료 결제(최소 5달러 크레딧 충전)의 불가피성 배경**

- OpenAI API 정책적 제한 (429 Credit Exceeded 에러): OpenAI는 웹 브라우저에서 사용하는 ChatGPT 무료 플랜과 달리, 외부 자동화 툴(Make)과 연동하는 API 서비스에 대해 엄격한 종량제 정책을 적용하고 있음
- 과거 제공되던 신규 계정 대상의 한시적 무료 크레딧이 전면 종료됨에 따라, 유효한 API Key를 발급받아 시스템을 매핑하더라도 계정에 최소 충전 금액인 $5.00의 크레딧이 선충전되어 있지 않으면 429 오류와 함께 동작 거부 일으키는 현상이 발생함

### 5.2 기술적 무료 대안 제시

- 대안 1: 무료 API를 제공하는 타사 LLM 레이어로 대체
    - Google AI Studio에서 제공하는 Gemini API 또는 Anthropic의 일부 무료 티어, 혹은 오픈소스 모델을 무료 API로 중개해 주는 Groq API 활용
    - Gemini Flash 모델의 경우 상용 수준의 성능을 제공하면서도 분당 요청 제한 범위 내에서는 무료 API 키를 발급해 주므로, OpenAI 모듈 대신 Gemini 모듈을 Make에 배치하면 비용 없이 동적 문장 생성 가능
- 대안 2: Make 내부 로직(Router 및 Switch 함수)을 활용한 의사 동적 멘트 처리
    - OpenAI 모듈을 제거하고 디스코드 모듈 전 단계에서 Make의 내장 함수 및 배열 기능을 활용하여 수십 개의 응원 문장을 미리 고정 배열로 리스트업해 둡니다. 그 후, 일정이 들어올 때마다 랜덤 함수를 통해 문장을 무작위로 뽑아 배핑
    - AI가 실시간으로 뇌를 굴려 짓는 문장은 아니지만 사용자 입장에서는 매번 디스코드 메시지가 무작위로 바뀌어 도달하므로 비용을 전혀 들이지 않고도 동적 응원 시스템의 시각적 효과를 유사하게 흉내 낼 수 있음
